# GB Empire Dashboard

Мини-дашборд заказов для тестового задания AI Tools Specialist.

Цепочка здесь такая:

`RetailCRM -> Supabase -> Next.js Dashboard -> Telegram`

## Deployment

Рабочий деплой на Vercel:

- [dashboard-nine-murex-68.vercel.app](https://dashboard-nine-murex-68.vercel.app/)

Это основной URL для проверки проекта.

## Что есть в проекте

- dashboard с заказами из Supabase
- сводные карточки и графики
- загрузка `mock_orders.json` в pipeline
- ручное добавление нового заказа через dashboard
- синхронизация заказов из RetailCRM в Supabase
- Telegram-уведомления для новых крупных заказов
- отдельная страница с отчётом: `/report`

## Важно перед проверкой

- Сейчас в таблице `orders` и в самом RetailCRM специально `0` заказов. Я оставил всё в чистом состоянии, чтобы сценарий проверки можно было пройти с нуля.
- Пример входных данных лежит в репозитории: `mock_orders.json`.
- Telegram-бот: `@GBEmpire_RetailCRM_bot`.
- Перед проверкой уведомлений нужно обязательно написать боту `/start`.

Без `/start` чат не попадёт в `telegram_users`, и отправлять уведомления будет просто некуда.

## Как этим пользоваться

### 1. Открыть dashboard

На главной странице есть:

- сводка по заказам
- графики по городам, источникам и статусам
- таблица заказов
- блок загрузки JSON
- форма ручного добавления заказа

Источник данных для dashboard: Supabase.

### 2. Загрузить mock JSON

На главной странице есть блок выгрузки заказов.

Что нужно сделать:

1. Ввести `pipeline secret`.
2. Выбрать `mock_orders.json` или свой JSON такого же формата.
3. Нажать `Запустить pipeline`.

Что делает pipeline:

1. принимает JSON
2. убирает дубли внутри входного файла
3. не загружает в RetailCRM заказы, которые там уже есть
4. загружает новые заказы в RetailCRM
5. забирает актуальные заказы из RetailCRM
6. синхронизирует их в Supabase через `upsert`

### 3. Добавить новый заказ вручную

На той же странице есть форма ручного добавления.

Через неё можно:

- заполнить контактные данные
- указать город и адрес доставки
- добавить один или несколько товаров
- отправить заказ в тот же pipeline

То есть ручной ввод и JSON-выгрузка идут по одной и той же логике, без второй отдельной серверной ветки.

### 4. Открыть report page

Есть отдельная страница:

- `/report`

Там я коротко показал, как собиралось приложение, где были проблемы и как я их раскладывал по шагам.

## Формат входных данных

Ориентир — `mock_orders.json`.

Pipeline принимает:

- либо JSON-массив заказов
- либо объект вида `{ "orders": [...] }`

Для ручной формы этот формат собирается автоматически.

## Telegram-уведомления

Уведомления приходят не на каждый заказ.

Текущая логика такая:

- уведомление отправляется только на **новый** заказ
- сумма должна быть **строго больше 50000 KZT**
- `50000` ровно не отправляется
- перед этим пользователь должен написать `/start` боту `@GBEmpire_RetailCRM_bot`

Ожидаемая цепочка:

`New order -> Supabase INSERT -> webhook -> /api/notifications/retailcrm-order -> Telegram`

## Безопасность

В проекте есть базовые меры защиты:

- Basic Auth на dashboard
- secret-проверка для pipeline route
- secret-проверка для webhook уведомлений
- server-only доступ к Supabase через `SUPABASE_SERVICE_ROLE_KEY`
- дополнительная защита Telegram webhook через secret token

Секреты и env-переменные сюда не дублирую. Файл с env передаётся отдельно.

## AI workflow

Эту задачу я собирал не одним большим запросом, а короткими итерациями. Сначала разбирался с данными и интеграциями, потом собрал dashboard, потом уже доводил pipeline, уведомления, защиту и подачу.

### Какие промпты я по сути давал

Ниже не буквальные цитаты, а суть запросов, которыми я реально вёл работу:

- разобрать, при каких условиях вообще должны приходить Telegram-уведомления и почему они не приходят после ручного добавления записи
- понять, достаточно ли просто вставить row в Supabase или нужен отдельный webhook на `INSERT`
- привести Python-скрипты к реальным env-переменным проекта и сделать так, чтобы они читали `.env.local`
- добавить передачу webhook secret через header и проверить, почему production отвечает `401`
- собрать полный pipeline внутри dashboard, чтобы можно было загружать `mock_orders.json` без ручных шагов вне приложения
- сделать ручное добавление нового заказа через UI на основе той же структуры, что и в `mock_orders.json`
- убрать риск дублей уведомлений, когда часть логики живёт в скрипте, а часть в webhook
- проверить проект перед деплоем: что закрыто, где нужен secret, где нужен service role key
- добавить `/report`, чтобы объяснить ход работы и решения не отдельным документом, а прямо внутри проекта
- привести report и dashboard к одному визуальному языку, потом развести светлую и тёмную темы

Если коротко, промпты были в духе: сначала найти точную причину, потом исправить, потом прогнать проверку.

### Где я реально застрял

Самые заметные затыки были такими:

1. Сначала казалось, что Telegram должен реагировать просто на новый заказ в базе, но на деле важен именно вызов webhook.
2. Ручной `INSERT` в Supabase не давал сообщения, потому что без Supabase Database Webhook это просто запись в таблицу.
3. Локально Telegram мог работать, а на Vercel нет. В итоге пришлось отдельно разбирать production env.
4. Был риск двойных уведомлений, если и Python-скрипт, и webhook отправляют одно и то же событие.
5. После добавления report страница отчёта и сам dashboard начали ощущаться как два разных продукта, это пришлось отдельно выравнивать.

### Как я это решал

Я шёл довольно приземлённо:

1. Сначала разложил реальную цепочку событий:

   `RetailCRM / import -> Supabase INSERT -> webhook -> Telegram`

2. Потом проверил код и зафиксировал факты:
   - уведомление идёт только на новый заказ
   - сумма должна быть строго больше `50000`
   - нужен подписанный пользователь в `telegram_users`

3. После этого стало ясно, что одного `INSERT` в таблицу мало — нужен настроенный Supabase Database Webhook.

4. Отдельно проверил технические причины, которые обычно ломают такие интеграции:
   - совпадает ли `RETAILCRM_WEBHOOK_SECRET`
   - тот ли `TELEGRAM_BOT_TOKEN` стоит на Vercel
   - есть ли пользователь в `telegram_users`
   - правильно ли сервер ходит в Supabase через `SUPABASE_SERVICE_ROLE_KEY`

5. Когда основная цепочка заработала, я уже собрал единый pipeline, чтобы и JSON upload, и ручное создание заказа шли через одну и ту же серверную логику.

6. В конце отдельно прошёлся по защите, интерфейсу и README, чтобы проект было не только можно запустить, но и нормально проверить.

### Почему отчёт есть и в README, и внутри приложения

README закрывает формальную часть тестового задания: по нему можно быстро понять, как это запускать, что проверять и как шла работа.

Страница `/report` — это уже более наглядная версия того же рассказа, но прямо внутри самого продукта. Мне показалось, что для такого тестового это нормальная и уместная форма подачи. Если нужно, это без проблем можно было бы свернуть в обычный PDF.
